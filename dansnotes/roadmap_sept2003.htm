VoxBo Roadmap, December 2003

Welcome to the VoxBo Roadmap, December 2003 Edition. The purpose of this document is twofold: to let VoxBo users know what's in the works and to make sure VoxBo developers are all on the same page. I'd be grateful for any feedback (by email or via discussion on voxbo-general), but bear in mind that this not an exhaustive list of our plans, and many feature requests and outstanding issues are not listed here. This is just an overview of the big picture stuff in the short- and medium term future of VoxBo, as we see it at the moment.

Recent and Imminent New Stuff

(These are things we have some concrete reason to be happy about.)

  • QT-based GLM designer - Dongbo Hu's first major contribution to VoxBo is available in the latest release (1.4). Although there are still things you can only do in the IDL version, we think for most purposes this is already preferable if you want to create a G matrix. If you want to edit or view an existing G matrix, it's much easier than playing with the IDL interface. The relevant executables are "glm" (the full GLM designer) and "gdw" (stand-alone G matrix designer). Now that we've moved it out of IDL, feature requests stand a much stronger chance of getting into the code sooner rather than later.
  • IDL-free GLM - Kosh Banerjee has ported a bunch of IDL-based GLM components to c, eliminating a big drain on IDL licenses. For labs on a budget, this might be a good time to think about how many of those IDL licenses you really want to renew.
  • vbview - Most VoxBo users are by now familiar with vbview, the replacement for voxsurf and maybe someday an alternative to the voxel surfer. vbview is the best way to edit multiple masks simultaneously in VoxBo right now, and as of release 1.5 handles more filetypes (including 4D), crashes less, and provides a kind of tri-view mode.
  • Unified, modular I/O - Kosh's IDLI code and my modular file format plug-ins together provide some powerful new tools for handling different file formats in VoxBo. If there's a format you'd like to see supported, please let us know.
  • Permutation Testing - Geoff's tools for permutation testing put VoxBo at (or at least near) the forefront of fMRI analysis packages for doing permutation-based analyses. I've written a bit of code myself, to handle conjoint permutation testing, cluster thresholding, and a few other twists. We consider the availability of permutation testing a major (and positive) shift in the direction of statistical analyses supported by VoxBo. If you aren't using permutation-based methods to threshold your maps, you may not be getting the most out of your data.
  • Quality Assurance - vbqa is a new(ish) program that goes through your data, finds things to warn you about (or things that you might want to check), and puts them all on one big web page. Although we can imagine lots of additional functionality that might be useful, the current incarnation is good enough to save you some grief. If you're not using it, give it a shot.
  • The mailing lists have been generally active and we hope helpful.
  • OS compatibility - VoxBo hasn't compiled under Solaris for quite a while now, but we hope that will change this summer. OSX compatibility is also on the horizon, which we expect will really help some labs take advantage of idle hardware. We're also hoping to have a more flexible scheduling system that's a little smarter about figuring out when your machine is available for jobs.
  • A VoxBo article, finally - it makes us happy when people cite the web page, but it will make us even happier when people cite the article. If all goes well, it will be in a Medline-indexed journal sometime in the next year.
  • Threshold setting for conjunctions using a permutation-based method is now available. As far as we know, this method is not available outside of VoxBo (although to be fair it would take all of five minutes to hack up for any package that already supports permutation). It's easy to use, and provides all the usual advantages of permutation testing. For more details, see our forthcoming (we hope) article. I can't resist mentioning that in the context of exploring the limits of the technique, we covered a parameter space that involved solving GLMs for around two billion voxels.
  • We continue to close in on some dramatically improved tools for structural methods. Jim Gee has a large codebase (see the ITK pages), and recently hired Loki der Quaeler, a postdoc with substantial software development expertise. Loki will be spending some time turning the raw code into useful tools for VoxBo users, and we expect to have some exciting new tools to announce this fall if not sooner.
  • VoxBo has a home - the newly formed Center for Functional Neuroimaging at Penn is the home of VoxBo development.
  • Completely free VoxBo (i.e., fully functional without IDL) is our number one goal, and we're getting close. See below for more details.
  • We're about to port all the documentation to DocBook format, which will facilitate making the VoxBo manual in various formats (esp. html and pdf). In the process of converting the old docs we also hope to update and expand things, to the point where we'll have a useful VoxBo manual again, incorporating both nuts and bolts information about the software and general help on imaging topics. In the meantime, I've started posting at least a few small documents on the web site. Of course, contributions from VoxBo users are always welcome. Post useful documents to voxbo-general and we'll fold them in.
  • The automation system we released last year, centered around the vbmakeprep script processor, turned out to be too much of a pain for most folks to use. I'm hoping to have a more intuitive version fairly soon.

    VoxBo Installation Service

    We've occasionally installed VoxBo for other labs, either at Penn or elsewhere. I've been thinking about formalizing this slightly, just by advertising that we're willing to help get it installed for anyone who needs help (and is willing to loan us their root password). This would be a good incentive for us to make the software easier to install, and might make other folks more comfortable trying it out.

    Completely Free VoxBo Update

    Yes, VoxBo still depends on IDL. We didn't even come close to getting rid of IDL by the end of last year, and this year looks improbable. But we have made a tremendous amount of progress. Here's an updated list of the big pieces that need replacement. Note that the list has shrunk dramatically, and is actually fairly short.

    • The voxel surfer. This is still the largest piece that hasn't been addressed. A lot of the functionality can be mimicked once we have a more modular system for generating statistical maps, but some pieces will take some more concerted effort.
    • The G matrix designer. Some functionality still hasn't been ported over from the IDL version, including some of the covariate transformations and the evaluation functions. Dongbo is working on these things (and more).
    • Smaller tools. I believe the three tools we're most desperate to have outside of IDL are close to reality. Dongbo has been working on a vector browser that will handle regular vectors (e.g., ref files) as well as movement parameters, replacing graphmoveparams. He's also working on new code to handle the 1/f parameter estimation and graphing. We should soon have a stand-alone smoothing tool, one way or another.
